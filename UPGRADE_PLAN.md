# 城市漫步 Agent 架构升级计划

> 基于当前代码库（单 ReAct 循环 + 4 字段 state + 滑动窗口截断）的现状分析，规划从"能用"到"好用"的升级路径。原始 8 条升级需求经过合并、重排后整理为 6 个阶段。

---

## 现状摘要

| 维度 | 当前实现 |
|------|---------|
| 决策模式 | 单一 ReAct 循环，所有请求走同一条路径 |
| 状态模型 | `AgentState` 4 字段：location / weather / candidates / conversation_history |
| 记忆 | 滑动窗口截断（20 candidates、40 history），无偏好记忆 |
| 工具 | 3 个：get_weather / search_places / resolve_location，顺序执行 |
| 输出形式 | 固定推荐 6 个散点，无路线概念 |
| System Prompt | 明确声明"不假设偏好、预算、行程等结构化字段" |

---

## 阶段一：Routing — 意图分流

**对应原计划：** #5

**为什么先做：** Routing 是所有后续 pattern 的入口。没有它，无法决定一个请求该走 ReAct、Plan-and-Solve 还是 REWOO。先做分流，后续每加一种新决策模式，只需在 router 里多注册一条路径。

**要改什么：**

- 在 `run_turn` 之前新增一个 **intent classifier** 层，将用户消息分为至少三类：闲聊 / 单点推荐 / 路线规划。后续阶段会追加更多类别（如"多方案"、"快速模式"）。
- 分类结果决定：走哪套 prompt、调用哪些工具、是否需要完整的工具循环。闲聊类可以直接走一次 LLM 调用，不进工具循环，省成本。
- Router 本身可以是一次轻量 LLM 调用（短 prompt + 结构化输出），也可以先用规则匹配兜底、命中不了再走 LLM。

**设计注意：**

- Router 的输出应该是一个枚举类型（如 `Intent`），下游按枚举分支处理，不要用字符串匹配。
- 为未来的路径类别预留扩展能力（比如用 registry pattern，类似现有 `tools/registry.py` 的设计）。

---

## 阶段二：Plan-and-Solve — 路线规划能力

**对应原计划：** #1

**前置依赖：** 阶段一（router 将"排路线"类请求分流到此路径）

**要改什么：**

- 新增一种决策模式：先让 LLM 生成一个 **结构化计划**（几点出发、途经顺序、穿插吃饭点），再按计划步骤逐一执行工具调用。与现有 ReAct "走一步看一步"的模式形成互补。
- `AgentState` 需要扩展，新增路线相关字段（如 `route_plan` / `time_constraints`）。这意味着 **system prompt 中"不假设行程字段"的声明必须同步修改**。
- 工具层面，`search_places` 可能需要支持"沿路线方向搜索"而非仅"以某点为圆心搜索"。可以考虑多次调用 search_places（不同中心点），或新增一个路线优化工具。

**设计注意：**

- Plan 和 Execute 分离：Plan 阶段只输出结构化方案（JSON），Execute 阶段按方案调工具。这样 Plan 可以给用户看、可以被修改（为阶段三的 Reflexion 做铺垫）。
- 路线不只是地点列表，还要有时间估算和交通方式提示。考虑是否需要引入新的外部 API（如步行导航）。

---

## 阶段三：Reflexion + 长期记忆 — 反馈学习与偏好沉淀

**对应原计划：** #2 + #7（合并）

**前置依赖：** 阶段二（有了路线规划，用户的"否决-修改"交互才有意义）

**为什么合并：** #2 的"记住用户否决了什么"和 #7 的"跨会话记住偏好"，本质上都是在一个当前不存在的"偏好/反馈记忆层"上读写。分开做会产生两套存储逻辑。

**要改什么：**

- 新增一个 **偏好记忆层**（独立于 conversation_history），分为两个层级：
  - **会话级反馈**：本轮对话中用户否决了什么、原因是什么（如"第三个太远了"）。用于本会话内后续推荐时规避。
  - **持久化偏好**：跨会话沉淀的用户画像（如"不爱酒吧""爱书店""走不动远路"）。会话结束时，从会话级反馈中提炼值得长期保留的偏好，写入持久化存储。
- `AgentState` 新增偏好相关字段；`SessionStore` 的持久化逻辑需要区分"会话状态"和"用户偏好"（偏好不应随会话清除）。
- System prompt 同步更新，告诉 LLM 如何读取和使用偏好信息。
- Reflexion 的核心机制：当用户否决推荐时，agent 把"否决内容 + 推断的原因"写入会话级反馈，下一次生成时在 prompt 中注入这些约束。

**设计注意：**

- 偏好提炼（从具体反馈到抽象偏好）可以用一次专门的 LLM 调用，在会话结束时运行。
- 偏好冲突处理：新反馈可能和旧偏好矛盾（用户以前说不爱日料，这次却专门要日料）。需要一个"时效性"机制，近期反馈优先。
- 持久化偏好的存储路径建议和 session 分开，放在用户级别（如 `.preferences/` 目录），不随 session 文件一起删。

## 阶段四：Evaluator-Optimizer — 输出质量自检

**对应原计划：** #4

**前置依赖：** 阶段二、三（有了路线规划和偏好记忆，评估才有足够的维度可检查）

**要改什么：**

- 在最终回复发给用户之前，新增一个 **评估步骤**：用一次独立的 LLM 调用（可以用更便宜的模型），按检查清单审核输出质量。
- 检查清单示例：是否回答了用户实际要的东西（6 个散点 vs 一条路线）、雨天是否全给了室内、推荐理由是否具体而非套话、路线顺序是否地理合理、是否违反了用户已知偏好。
- 评估不通过 → 把评估意见作为 feedback 注入 prompt，让生成模型重做一次（最多重做 1-2 次，避免无限循环）。

**设计注意：**

- Evaluator 的 prompt 要和 Generator 的 prompt 独立，避免同一个模型"自己评自己"时的盲点。如果条件允许，用不同的模型或不同的 temperature。
- 评估结果应该是结构化的（通过/不通过 + 各维度的具体意见），不要用自由文本。
- 性能预算：评估增加一次 LLM 调用，重做再加一次。对于延迟敏感的场景，考虑做成可配置的开关。

## 阶段五：并行执行框架 — 多方案生成与复杂请求分解

**对应原计划：** #3 + #6（合并）

**前置依赖：** 阶段一（router 决定何时触发并行）

**为什么合并：** #3（多方案并行）和 #6（Orchestrator-Workers）的执行机制几乎相同——都是"拆成多个独立子任务 → 并发执行 → 聚合结果"。区别仅在上游语义：#3 是"同一个需求出多个变体"，#6 是"一个复杂需求拆成多个子需求"。底层可以共享一个并发调度框架。

**要改什么：**

- 新增一个 **并行调度层**，能够接收一组独立的子任务描述，为每个子任务分别构造 prompt + 工具集，并发调用 LLM，最后由一个"聚合器"将子结果合并为最终输出。
- 当前 `run_turn` 里的 tool dispatch 是顺序 for 循环，需要改为支持 `asyncio.gather` 或类似机制的并发执行。但注意：同一个子任务内部的工具调用仍然是顺序的（ReAct），并发只发生在子任务之间。
- Router（阶段一）需要新增分类：识别出"多方案请求"和"可拆分的复杂请求"，分别构造子任务。

**两种模式的区分：**

| | 多方案模式（原 #3） | 任务分解模式（原 #6） |
|---|---|---|
| 触发 | "给我雨天和晴天两种方案" | "找适合老人和小孩的地方" |
| 子任务生成 | 同一查询 × 不同约束 | 一个查询 → LLM 拆成多个子查询 |
| 聚合方式 | 平行呈现，用户选择 | 交叉过滤/合并，输出统一结果 |

**设计注意：**

- 子任务数量需要上限（建议 2-4），避免 LLM 过度拆分导致成本失控。
- 聚合器可能需要一次 LLM 调用来做智能合并（尤其是任务分解模式下的地理重叠筛选），不能简单拼接。
- 并发调用外部 API（腾讯地图）时注意 rate limit，当前的日调用额度限制在并发场景下更容易触顶。

## 阶段六：REWOO — 快速出方案模式

**对应原计划：** #8

**前置依赖：** 阶段一（router 将"赶时间"请求分流到此路径）、阶段四（决定 REWOO 是否需要评估）

**要改什么：**

- 新增一种决策模式：LLM 一次性规划出所有需要的工具调用及其参数（生成一个工具调用计划），然后批量执行所有工具调用（无中间观察），最后把全部结果一次性交给 LLM 做最终合成。
- 与 ReAct 的对比：ReAct 是"调一个工具 → 看结果 → 决定下一步"，REWOO 是"先全想好 → 一口气调完 → 最后统一总结"。REWOO 更快（LLM 调用次数少），但缺乏中间纠错能力。

**与 Evaluator（阶段四）的协调：**

- 这是此前分析中提到的主要张力点。建议方案：REWOO 模式下使用 **轻量版评估**——只做格式和完整性检查（是否有足够的地点、是否回答了用户问题），跳过深度质量评估。不通过时 fallback 到标准 ReAct 路径，而非在 REWOO 内部重做。

**设计注意：**

- 工具调用计划需要是结构化的（JSON 数组），每个元素包含工具名和参数。LLM 需要在没有中间结果的情况下预测所有参数，这对 prompt 设计有要求。
- 某些工具有依赖关系（比如先 resolve_location 拿到经纬度，再 search_places 用这个经纬度）。REWOO 需要支持 **变量引用**：计划中后续步骤可以引用前序步骤的输出字段。执行时按依赖顺序执行，但跳过"观察 → 再决策"的循环。

## 需要贯穿各阶段的横切改动

以下改动不属于某个特定阶段，而是随着升级推进需要持续调整的部分：

**System Prompt 演进：** 当前 prompt 是为单一 ReAct + 6 地点推荐写的。从阶段一开始就需要根据 intent 类型使用不同的 prompt（或 prompt 片段）。建议将 `prompts.py` 从单一常量改为按 intent 组装的 prompt builder。

**AgentState 扩展：** 阶段二加路线字段，阶段三加偏好字段。建议保持 pydantic model 的设计，但改为分层结构（核心字段 + 可选扩展字段），避免所有模式都要承担全部字段的序列化开销。

**Guardrails 扩展：** 输入护栏可能需要识别新的意图类型；输出护栏的检查维度随路线规划和多方案模式增加。评估器（阶段四）在某种意义上是一个"智能版输出护栏"，需要想清楚它和现有 `output_filter.py` 的职责边界。

**测试策略：** 当前项目没有测试。升级过程中，建议至少为 router 的分类准确性和 evaluator 的检查清单写端到端测试，否则后续阶段的改动很容易破坏前序阶段的行为。
